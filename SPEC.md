# opencode-k8s-sandbox — Implementation Spec

## 1. Purpose

Let two or more people share a single OpenCode server and have each session
run in its own isolated Kubernetes pod, with any web server the agent starts
inside that pod reachable from a browser at a predictable URL — without
provisioning a Kubernetes Ingress or Service per session.

Two independently deployable components:

- **`router`** (Rust) — a long-running TCP proxy that resolves
  `{port}-{sandbox-id}.{base-domain}` to a pod IP/port and splices the
  connection. One Ingress, ever, points at this.
- **`plugin`** (TypeScript, OpenCode plugin) — hooks OpenCode's session
  lifecycle to create/destroy a pod per session, and routes file/bash tool
  calls into that pod instead of the local filesystem.

Both are cluster-agnostic: no assumptions about which ingress controller,
CNI, or cloud the operator runs. All cluster-specific values are config.

## 2. Non-goals (explicit exclusions for v1)

- No auth/authz layer in either component. Access control is delegated to
  the network (e.g. a private tailnet). Document this loudly in the README
  — anyone deploying this on a network without equivalent isolation needs to
  put an auth proxy in front of both the OpenCode server and the router.
- No HA/multi-replica router. Single replica is fine; if it restarts, the
  in-memory pod index rebuilds from a fresh k8s watch (LIST + WATCH) in
  under a second.
- No protocols other than HTTP/1.1 and WebSocket. No raw
  TCP passthrough for non-HTTP services.
- No horizontal scaling of sandbox pods (one pod = one session, not a pool).
- No resource-quota policy engine — just static requests/limits from config.

## 3. Naming and shared conventions

These two components only agree on a contract via Kubernetes object
metadata — they never talk to each other directly.

- **Sandbox ID**: an 8-character lowercase hex string, generated by the
  plugin per OpenCode session (`crypto.randomBytes(4).toString("hex")`).
  OpenCode's own session IDs are not guaranteed to be short or DNS-label
  safe, so never use them directly in object names or hostnames — keep a
  session-ID → sandbox-ID mapping inside the plugin's runtime state.
- **Pod name**: `opencode-sbx-{sandbox-id}`
- **Pod labels** (router watches on these):
  - `app.kubernetes.io/managed-by: opencode-k8s-sandbox`
  - `opencode.dev/sandbox-id: {sandbox-id}`
- **Pod annotation**: `opencode.dev/session-id: {full opencode session id}`
  (for operator debugging only; router never reads this)
- **Hostname pattern**: `{port}-{sandbox-id}.{base-domain}`
  e.g. `5173-a1b2c3d4.opencode.example.com`
- **Base domain** is operator-supplied config, identical value required on
  both components (router validates/serves it, plugin doesn't need it
  except to print convenience URLs in tool output/logs).

## 4. Component: `router` (Rust)

### 4.1 Responsibilities

1. Maintain a live, in-memory index of `sandbox-id → pod IP` by watching
   Pods with label `opencode.dev/sandbox-id` in the configured namespace(s).
2. Accept plain HTTP TCP connections (TLS already terminated upstream by
   the cluster's ingress controller).
3. Parse the `Host` header off the wire, extract `{port}` and
   `{sandbox-id}`, look up the pod IP, and proxy the raw TCP stream to
   `{pod-ip}:{port}` — no HTTP re-framing, so HTTP/1.1 keep-alive, SSE, and
   WebSocket upgrades all work without protocol-specific code.
4. Expose a separate health/metrics listener, isolated from the proxy hot
   path.

### 4.2 Crate layout

```
router/
  Cargo.toml
  src/
    main.rs          # wiring: config, watcher, both listeners
    config.rs         # env-driven config struct
    index.rs           # sandbox-id -> PodIp reflector wrapper
    proxy.rs            # accept loop, header sniff, splice
    health.rs           # /healthz, /metrics (axum or hyper, separate port)
  Dockerfile             # multi-stage, musl target, distroless final stage
  README.md
```

### 4.3 Dependencies

- `tokio` (multi_thread, small worker count is fine)
- `kube` + `k8s-openapi` — use `kube::runtime::watcher()` +
  `kube::runtime::reflector` to build the index; do not poll the API per
  request
- `httparse` — header-only parse of the buffered bytes to read `Host`
- `tracing` + `tracing-subscriber`
- `axum` (or bare `hyper`) — health/metrics listener only, on a distinct
  `SocketAddr` from the proxy listener
- Metrics: expose Prometheus text format at `/metrics` on the health
  listener — at minimum: active connection count, total connections,
  lookup-miss count, bytes proxied

### 4.4 Behavior spec

**Startup**
- Read config (see 4.5).
- Build k8s client (in-cluster config by default; fall back to
  `KUBECONFIG`/default kubeconfig for local dev).
- Start the reflector watch on Pods matching the label selector
  `opencode.dev/sandbox-id` (and namespace, if `ROUTER_NAMESPACE` is set;
  otherwise cluster-wide with an RBAC `ClusterRole`).
- Start the health listener.
- Start the proxy listener.

**Per connection**
1. Accept TCP connection.
2. Read into a growable buffer until `\r\n\r\n` is found (headers-complete)
   or a max header size is exceeded (reject with `400` written directly as
   raw bytes, then close — no HTTP library needed for this one path).
3. Run `httparse::Request::parse`, extract the `Host` header value.
4. Validate/parse hostname with a regex or manual split:
   `^(\d{1,5})-([0-9a-f]{8})\.` — capture `port` and `sandbox_id`. If it
   doesn't match, close with a plain-text `400`.
5. Look up `sandbox_id` in the reflector-backed index. If missing, close
   with a plain-text `502` ("sandbox not found or not ready").
6. Dial `{pod_ip}:{port}` with a short connect timeout (e.g. 3s). If the
   dial fails (server hasn't started listening yet inside the pod), close
   with `502`.
7. Write the already-buffered header bytes to the backend connection, then
   `tokio::io::copy_bidirectional(&mut client, &mut backend)` for the rest
   of the connection's life.

**Pod lifecycle correctness**
- On pod delete/replace, the reflector watch updates the index
  automatically — no manual TTL/expiry logic needed in the proxy itself.
- Only route to pods with `status.phase == Running` and a non-empty pod IP;
  filter this in the reflector's `Store` accessor, not at watch-registration
  time.

### 4.5 Config (all via environment variables)

| Var | Required | Default | Notes |
|---|---|---|---|
| `ROUTER_PROXY_ADDR` | no | `0.0.0.0:8080` | plain HTTP listener |
| `ROUTER_HEALTH_ADDR` | no | `0.0.0.0:9090` | `/healthz`, `/metrics` |
| `ROUTER_NAMESPACE` | no | all namespaces | restrict the watch |
| `ROUTER_LABEL_KEY` | no | `opencode.dev/sandbox-id` | override for multi-tenant reuse |
| `ROUTER_LOG` | no | `info` | tracing filter string |
| `ROUTER_CONNECT_TIMEOUT_MS` | no | `3000` | backend dial timeout |
| `ROUTER_MAX_HEADER_BYTES` | no | `8192` | guards the header-sniff buffer |

### 4.6 Deployment artifacts

- `Dockerfile`: multi-stage, `x86_64-unknown-linux-musl` target,
  `FROM gcr.io/distroless/static-debian12` (or `scratch` + static CA
  bundle if TLS egress is ever needed) final stage.
- `deploy/` (plain Kustomize base, not Helm — matches the rest of the
  cluster's tooling style referenced by the operator):
  - `deployment.yaml` — single replica, resource requests/limits
  - `service.yaml` — ClusterIP, `port: 8080`
  - `serviceaccount.yaml` + `clusterrole.yaml` (get/list/watch on `pods`)
    + `clusterrolebinding.yaml`
  - `ingress.yaml.example` — commented template showing the wildcard host
    rule the operator adds once, pointed at whatever ingress controller
    they run (do not hardcode Traefik-specific CRDs like `IngressRoute` in
    the shipped manifest; document the Traefik variant separately in
    README as an example)

## 5. Component: `plugin` (TypeScript, OpenCode plugin)

### 5.1 Responsibilities

1. On session creation, provision an isolated pod for that session.
2. Route the built-in `bash`, `read`, `write`, `edit` (and any other
   filesystem/exec) tools into that pod instead of the local host.
3. On session end or idle timeout, tear the pod down (and its PVC, unless
   configured to persist).
4. Surface the sandbox's access URLs back to the user in tool output when
   relevant (e.g. after a `bash` call that starts a dev server).

### 5.2 Package layout

```
plugin/
  package.json
  src/
    index.ts            # exports the plugin factory OpenCode loads
    config.ts             # zod schema + env/config-file loader
    k8s/
      client.ts             # @kubernetes/client-node wrapper
      podSpec.ts              # builds the Pod (and optional PVC) manifest
      exec.ts                  # exec-in-pod primitive (shared by all tool overrides)
    sessionStore.ts       # session-id -> { sandboxId, podName, createdAt, lastActiveAt }
    tools/
      bash.ts                 # override: exec inside pod
      fileOps.ts               # override: read/write/edit via exec (cat / heredoc, base64 for binary)
    hooks.ts               # session.created / session.deleted / session.idle wiring
  README.md
```

### 5.3 Dependencies

- `@kubernetes/client-node`
- `zod` (matches OpenCode's own tool-definition convention)
- No HTTP framework needed — the plugin never listens on a port; it only
  makes outbound calls to the k8s API server.

### 5.4 Hook wiring (`hooks.ts`)

- **`session.created`**
  1. Generate `sandboxId` (8-char hex).
  2. Build and apply the Pod manifest (5.5) via `k8s/podSpec.ts`.
  3. Optionally build and apply a PVC if `persistWorkspace` is enabled in
     config, and mount it into the pod at the working directory.
  4. Poll pod status (bounded retries, e.g. up to 30s) until
     `status.phase == Running` before returning, so the first tool call
     doesn't race pod startup.
  5. Store `{ sandboxId, podName, createdAt, lastActiveAt }` keyed by
     OpenCode's session ID in `sessionStore`.
  6. Log the sandbox's base URL pattern
     (`https://{port}-{sandboxId}.{baseDomain}`) via `client.app.log()`.
- **`session.deleted`**
  1. Look up the sandbox record; if none, no-op.
  2. Delete the Pod (and PVC, unless `persistWorkspace`).
  3. Remove the session from `sessionStore`.
- **`session.idle`** (if OpenCode exposes an idle hook; otherwise implement
  a `setInterval` sweep inside the plugin comparing `lastActiveAt` against
  `idleTimeoutMinutes` from config)
  1. Same teardown as `session.deleted`.
- **`tool.execute.before` / `tool.execute.after`**
  - Update `lastActiveAt` for the session on every tool call, for the idle
    sweep above.

### 5.5 Pod spec (`podSpec.ts`)

- Single container, image from config (`sandboxImage` — operator supplies
  an image containing `git`, language toolchains, and anything the agent
  needs; this project does not ship or opinionate on that image beyond a
  documented minimal example in `docs/`).
- `command`/`args`: keep the container alive (e.g. `["sleep", "infinity"]`)
  — all actual work happens via `exec`, not via the container's entrypoint.
- Init container (or a startup exec call from the plugin, either is fine —
  init container is cleaner) that runs `git clone {repoUrl}` into the
  working directory, using the repo URL passed at session-creation time
  (see 5.7).
- Labels/annotations as defined in section 3.
- `resources.requests`/`limits` from config.
- No `containerPorts` declared — the router connects directly to the pod
  IP on whatever port the agent's process binds, so nothing needs to be
  pre-declared.
- If `persistWorkspace`: mount a PVC at the working directory;
  `storageClassName` from config.

### 5.6 Exec primitive (`k8s/exec.ts`)

One shared function, used by every tool override:

```ts
execInPod(podName: string, namespace: string, command: string[]): Promise<{ stdout: string; stderr: string; exitCode: number }>
```

Built on `@kubernetes/client-node`'s `Exec` class (SPDY-based exec stream).
All tool overrides — `bash`, `read`, `write`, `edit` — funnel through this
one primitive rather than each reimplementing exec plumbing:

- `bash` → command passed straight through.
- `read` → `cat {path}`, capture stdout.
- `write` → pipe content in via stdin using a heredoc or base64 encode +
  `base64 -d > {path}` for binary-safety.
- `edit` → read, patch in-process (same diff/patch logic OpenCode's local
  edit tool already uses), write back.

### 5.7 Session-creation input (repo URL)

OpenCode's session creation doesn't natively carry "which repo". The plugin
uses a **manifest-based approach** with automatic project discovery:

1. **Manifest config**: Operator defines a `repos` map in plugin config
   mapping project names to git URLs.
2. **Server-side cloning**: On plugin init, repos are shallow-cloned into
   subdirectories under the worktree (e.g., `{worktree}/repos/{name}/`).
3. **Project discovery**: OpenCode lists child directories as available
   projects in the UI. Users select a project by opening a session in it.
4. **Session creation**: When `session.created` fires, the plugin detects
   the current directory via `client.path.get()`, maps it to a repo URL,
   and passes the URL to the pod's init container for cloning.

This approach:
- Requires no custom UI — leverages OpenCode's native project listing
- Supports multiple repos from a single OpenCode server
- Re-clones repos on plugin init to pick up upstream changes
- Creates pods with the selected repo cloned via init container (~5-15s cold start)

### 5.8 Config (`config.ts`, zod-validated)

| Key | Required | Default | Notes |
|---|---|---|---|
| `namespace` | yes | — | where sandbox pods are created |
| `sandboxImage` | yes | — | operator-supplied image (needs nix + direnv) |
| `repos` | yes | — | map of project names to git URLs |
| `repoBaseDir` | no | `repos` | subdirectory under worktree for clones |
| `baseDomain` | yes | — | must match router's serving domain |
| `resources.requests.cpu` / `.memory` | no | `250m` / `256Mi` | |
| `resources.limits.cpu` / `.memory` | no | `2` / `2Gi` | |
| `persistWorkspace` | no | `false` | keep PVC across pod restarts |
| `storageClassName` | no (required if `persistWorkspace`) | — | |
| `idleTimeoutMinutes` | no | `60` | |
| `podStartupTimeoutSeconds` | no | `30` | |

Example config:
```json
{
  "namespace": "opencode-sandbox",
  "sandboxImage": "opencode-sandbox:latest",
  "repos": {
    "frontend": "git@github.com:org/frontend.git",
    "api": "git@github.com:org/api.git"
  },
  "baseDomain": "opencode.example.com"
}
```

Loaded from OpenCode's plugin config block (per OpenCode's plugin config
convention) with env-var overrides:
- `SANDBOX_NAMESPACE` → `namespace`
- `SANDBOX_IMAGE` → `sandboxImage`
- `SANDBOX_REPOS` → `repos` (JSON string)
- `SANDBOX_REPO_BASE_DIR` → `repoBaseDir`
- `SANDBOX_BASE_DOMAIN` → `baseDomain`

## 6. Cluster prerequisites (one-time, operator-performed, not automated by either component)

1. Wildcard DNS record: `*.{baseDomain}` → the ingress IP.
2. Wildcard TLS cert covering `*.{baseDomain}` (DNS-01 challenge, since
   there's no single reachable host for HTTP-01) issued however the
   operator's cluster already issues certs.
3. One Ingress object routing `*.{baseDomain}` → the router Service. Ships
   as an example manifest, not applied automatically by either component.
4. RBAC applied for the router (`ClusterRole`: get/list/watch pods) and for
   the plugin's service account if OpenCode itself runs in-cluster
   (`ClusterRole` or namespaced `Role`: create/get/delete on pods, and
   pods/exec, and optionally persistentvolumeclaims).

## 7. Acceptance criteria

- [ ] Creating an OpenCode session results in a Running pod labeled
      `opencode.dev/sandbox-id=<id>` within `podStartupTimeoutSeconds`.
- [ ] `bash`/`read`/`write`/`edit` tool calls during that session operate
      on the pod's filesystem, verified by a file written via the `write`
      tool being visible via `kubectl exec`.
- [ ] Ending the session (or idle timeout) deletes the pod within a few
      seconds.
- [ ] Starting an arbitrary process inside the pod bound to an arbitrary
      port (test with at least three different ports, e.g. 8080, 8888,
      5173) is reachable at `https://{port}-{sandboxId}.{baseDomain}` from
      a browser on the tailnet.
- [ ] A dev server with HMR (e.g. Vite) works end-to-end through the
      router, including the WebSocket upgrade — verified by editing a file
      and observing a live-reload in the browser without a manual refresh.
- [ ] Killing and restarting the router pod re-establishes correct routing
      for all still-running sandboxes within a few seconds, without
      requiring any sandbox pod to restart.
- [ ] Only one Ingress object exists in the cluster regardless of how many
      concurrent sessions are open.

## 8. Suggested build order

1. **Router skeleton**: static config + reflector watch + `/healthz`.
   Verify the index populates correctly against a couple of manually
   `kubectl run` pods carrying the right label, no proxy logic yet.
2. **Router proxy path**: header sniff + splice, tested against the manual
   pods from step 1 running a plain `python -m http.server`.
3. **Plugin pod lifecycle only**: `session.created`/`session.deleted`
   creating/destroying pods, no tool overrides yet — verify via `kubectl
   get pods -w` while creating/ending OpenCode sessions.
4. **Plugin tool overrides**: wire `bash`/`read`/`write`/`edit` through the
   shared exec primitive.
5. **End-to-end**: wildcard DNS + cert + Ingress wired up for real, full
   flow from "create session" → "agent starts a dev server" → "browser
   loads it over the tailnet."
6. **Packaging/polish**: Dockerfile + Kustomize base for the router, npm
   packaging for the plugin, README for both, docs for the "bring your own
   ingress controller" step and the "bring your own sandbox image" step.
